### [F-008] M-001 — Admin role gate on bulk Excel import
- File: [app/api/import/route.ts:42](app/api/import/route.ts:42)

### [F-009] M-002 — Batched deletes & inserts on import (N+1 → fixed-cost)
- File: [app/api/import/route.ts:122-260](app/api/import/route.ts:122)
- Was: per-company → 1 lookup + 3 deletes + N×3 inserts. A 10-company × 5-row workbook ≈ 150 round-trips.
- Now: single `IN(...)` company lookup up front; per company, a single `insert([...rounds])` followed by single batched `insert([...investments])` and `insert([...cap_table])`, threaded by the returned round IDs (PostgREST preserves input order). Same workbook now ≈ 10–13 round-trips total. Also adds a guard that aborts a company if `insertedRounds.length` drifts from input, so investments can never be written without matching rounds.

### [F-010] M-003 — Removed `CRON_SECRET` fallback from send-report

## IMPROVEMENT IDEAS

1. **Move the rate limiter to Upstash/KV.** [lib/rate-limit.ts](lib/rate-limit.ts) is per-process, so a determined attacker amplifying across Vercel cold instances can still exceed the cap. The interface (`checkRateLimit(key, max, windowMs) → string | null`) is intentionally identical to what an Upstash sliding-window helper would expose, so swapping the implementation should be a one-file change.

2. **Replace the `import` route loop with one RPC per company.** F-009 made it batched-3-inserts-per-company. The next step is a single `import_company_rows(p_company_id, p_rows jsonb)` Postgres function so the whole company's replace-cycle is one transaction. The pattern already exists in `convert_safe` (above) — copy the shape once the safes refactor lands.

3. **Migrate logos to `next/image` once you decide the trust boundary.** Right now `logo_url` is "any URL the user pastes in". Either constrain it at write time (validate and proxy through Supabase storage) or accept the optimization-vs-trusted-host tradeoff and broaden `images.remotePatterns`.

4. **Adopt a `useAsyncWithToast` hook.** The composition of `useTransition` + `useToast` to handle "trigger an action, optimistically render, toast on failure" recurs across `ContactTasks`, `ProfileClient`, and most form-y components. The Q5 audit flagged it as an anti-pattern; building the hook is a 30-minute job once the team agrees on the loading/error/retry contract.

5. **ESLint rule for dark-mode pairs.** The remaining dark-mode misses are mechanical: "if you write `text-*-800` / `bg-white` / `text-slate-300`, you must also write a `dark:` sibling". A custom rule (or even a simple repo-grep CI step) would catch the next regression at PR time without anyone having to spot it in screenshots.
