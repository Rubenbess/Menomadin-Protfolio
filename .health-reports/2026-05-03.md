## FIXED ISSUES

### [F-001] Reminder email send failures silently reported as success
- File: app/api/cron/reminders/route.ts (line 74)
- Problem: `resend.emails.send()` was awaited but its return value was never checked. A failed email delivery (bad API key, rate limit, network error) would still return `{ sent: N }` to the cron caller, making it impossible to detect or alert on delivery failures.
- Fix applied: Destructure `{ error: sendError }` from the send call; return HTTP 500 with the error message if it is truthy.

### [F-002] Microsoft OAuth token storage failure silently redirects to success
- File: app/api/auth/microsoft/callback/route.ts (line 82)
- Problem: `supabase.from('email_integrations').upsert(...)` was awaited without capturing or checking the error. If the database write failed (RLS violation, schema mismatch, connection error), the user was redirected to `/settings/email-scanner?success=connected` even though their token was never stored — leaving their account in a permanently broken state.
- Fix applied: Destructure `{ error: upsertError }` from the upsert; redirect to `?error=store_failed` if it is truthy, preventing false-success UI.

### [F-003] XIRR non-convergence emitted console.warn in production
- File: lib/calculations.ts (line 122)
- Problem: `console.warn('calcXIRR: did not converge...')` was left in a production code path. This fires on every IRR calculation that doesn't converge (e.g. early-stage portfolios with all outflows and no realizations), polluting server logs and leaking internal cash flow data shapes to browser consoles on the client.
- Fix applied: Removed the console.warn; the function already returns `null` correctly on non-convergence, so no behavior change.

### [F-004] useListFilters recomputed filter result on every render
- File: hooks/useListFilters.ts (line 54)
- Problem: `filteredData` was defined as `useCallback` (memoizes the *function reference*) but was immediately called as `filteredData()` at module level. This means the filter+sort computation ran on **every render** regardless of whether dependencies changed — negating the memoization entirely. On large deal or company lists this causes noticeable UI jitter.
- Fix applied: Replaced `useCallback` + inline call with a single `useMemo` that computes and returns the filtered array directly. The `filteredData` export is removed (no callers depended on the callable form).

## MANUAL ACTION REQUIRED

### [M-001] Import route silently drops update/delete errors
- Severity: High
- Category: Bug
- File: app/api/import/route.ts (lines 135–140, 166–168)
- Problem: `.update()` and `.delete()` calls during company re-import are not error-checked. If a delete of existing investment data fails mid-import, the route continues inserting new rows, leaving the database in a partially-updated state with duplicates. There is no rollback.
- Recommended fix: Destructure and check `error` on every mutating Supabase call inside the import loop. Consider wrapping the entire per-company mutation sequence in a Postgres transaction via RPC or a batch upsert strategy.

### [M-002] Import route counter logic hides partial failures
- Severity: Medium
- Category: Bug
- File: app/api/import/route.ts (lines 213, 223)
- Problem: `if (!invErr)` / `if (!ctErr)` increment success counters only on success, but failed rows are silently skipped with no record of which rows failed or why. The response's `warning:` field is populated but the UI may not display it.
- Recommended fix: Collect failed row identifiers into an array and include them in the response alongside the success count. Display warnings in the import UI component.

### [M-003] SAFE conversion division by zero when nextPreMoney is 0
- Severity: Critical
- Category: Bug (Financial Data Integrity)
- File: lib/calculations.ts (lines 176–177)
- Problem: `calcSafeConversion` computes `newInvUnits = roundRaise / nextPreMoney`. If `nextPreMoney` is 0 (user enters a hypothetical round with $0 pre-money), the result is `Infinity`, which propagates to `ownershipPct` and `sharesValue` displayed in the SAFE modelling UI. Similarly, if `effectiveVal` resolves to 0, `safeUnits` becomes `Infinity`.
- Recommended fix: Add a guard at the top of `calcSafeConversion`: `if (!nextPreMoney || nextPreMoney <= 0 || !effectiveVal || effectiveVal <= 0) return null` (or a typed error result). All callers already handle a null/error return.

### [M-004] Dashboard fetches all portfolio tables without pagination
- Severity: High
- Category: Performance
- File: app/(protected)/dashboard/page.tsx (lines 55–73)
- Problem: `rounds`, `investments`, `cap_table`, `reserves`, `safes`, and `legal_entities` are fetched with no `.limit()` or `.range()`. As the portfolio grows past ~500 investments this query will become a slow full-table scan on every dashboard load. The page is `force-dynamic` so there is no caching.
- Recommended fix: For the dashboard aggregate view, consider a server-side summary RPC function that returns pre-aggregated metrics rather than raw row data. Alternatively, add Supabase server-side caching (via `supabase-cache-helpers` or ISR with revalidation).

### [M-005] Company detail page swallows all query errors
- Severity: Medium
- Category: Bug
- File: app/(protected)/companies/[id]/page.tsx (lines 16–44)
- Problem: `Promise.all` destructures only `{ data: ... }` from all 13 parallel queries, discarding every `error`. A failed fetch (RLS block, network timeout, bad foreign key) silently passes `null` data to the client component as `[]`, showing empty tables with no indication of failure.
- Recommended fix: Destructure errors and throw/log them, or add a helper that wraps each query and rethrows on error so Next.js can render the error boundary.

### [M-006] Missing error check on contact delete in companies action
- Severity: Medium
- Category: Bug
- File: actions/companies.ts (line 46)
- Problem: `await supabase.from('contacts').delete().eq('company_id', companyId)` has no error handling. If the delete fails (e.g. RLS, FK constraint), the action continues and overwrites the company record with stale contact data silently.
- Recommended fix: Destructure `{ error }` and return early with an error response if the delete fails.

### [M-007] N+1-style nested loop in import route
- Severity: Medium
- Category: Performance
- File: app/api/import/route.ts (lines 120–226)
- Problem: Each company in the import is processed sequentially in a `for...of` loop with individual Supabase calls per row. A 100-company import triggers 200–400 sequential round trips. This will timeout on Vercel's 10-second function limit for medium-sized portfolios.
- Recommended fix: Batch all inserts per table using `.upsert([...rows])` in a single call after the loop, rather than one call per row.

### [M-008] hardcoded stage and strategy lists in CompanyForm

The memory index already notes a known mismatch between capitalized DB values ("Prospecting", "Due Diligence") and stale lowercase references elsewhere.

### [M-010] Missing empty state in filtered companies list

### [M-011] PDF generation logic embedded in API route handler

### [M-012] IRR/XIRR shows "N/A" with no user explanation

- Recommended fix: Add a tooltip or info icon explaining: "IRR requires at least one positive cash flow (realized return or current portfolio value) alongside investment outflows."

## IMPROVEMENT IDEAS

1. **Introduce a Supabase query wrapper with unified error handling.** Every server action and page currently has bespoke `if (error) return { error: error.message }` patterns. A small `query<T>()` wrapper that throws on error would eliminate the class of "forgotten error check" bugs (M-001, M-002, M-005, M-006) by making success the only non-throwing path.

2. **Replace full-table dashboard fetches with a portfolio summary RPC.** A single Postgres function returning pre-aggregated totals (total invested, current value by company, MOIC) would reduce dashboard load from 7 parallel full-table queries to 1 RPC call — critical as the portfolio scales past 50 companies.

3. **Add a constants file for pipeline stages and strategies.** The memory index already flags a known mismatch between DB enum values and UI references. Centralizing these in `lib/constants.ts` and linting for raw string usage would prevent the next mismatch from reaching production silently.

4. **Add server-side error boundaries to all data-heavy protected pages.** Next.js 14 `error.tsx` files next to each page would convert silent data failures into user-visible error states with retry buttons — currently there is no error boundary on `/dashboard`, `/companies/[id]`, or `/pipeline`.

5. **Surface import warnings in the UI.** The import route already returns a `warnings` array in its response, but if the client component discards it, partial import failures are invisible to the user. Displaying a dismissible warning banner after import would make data integrity issues actionable.
