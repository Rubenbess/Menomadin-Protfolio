## FIXED ISSUES

### [F-001] Missing pagination limit on `legal_entities` query
- File: `app/(protected)/analytics/page.tsx:40`
- Problem: Five of the six parallel Supabase queries on the analytics page used `.limit(FULL_LIST_FETCH_LIMIT)`, but the `legal_entities` query was unbounded. Defensive bounds on every list query.
- Fix applied: Appended `.limit(FULL_LIST_FETCH_LIMIT)`.

### [F-002] `any[]` typing in analytics layer
- File: `lib/analytics-utils.ts` (5 functions); `app/(protected)/analytics/AnalyticsClient.tsx:21-22`
- Problem: `calculatePortfolioMetrics` etc. and the `AnalyticsClient` props were typed `any[]`. Untyped financial-data inputs are exactly the surface where a typo silently produces zero.
- Fix applied: Added a shared `CompanyWithMetrics` interface and narrowed every call site.

### [F-003] convertSafe — best-effort rollback when cap-table insert fails (was M-001)
- File: `actions/safes.ts:55-118`
- Problem: SAFE row was marked `converted` first, then cap-table row inserted. If the insert failed the SAFE was permanently marked converted with no cap-table entry — silent corruption of waterfall and MOIC.
- Fix applied: After a cap-table insert error, the action now reverts the SAFE row back to `unconverted` / `converted_round_id: null`. If the rollback itself fails, a combined error is returned naming both errors and the SAFE id so an operator knows manual cleanup is needed. Caveat: still not a true DB transaction — interleaving writes from another caller could still corrupt state. Proper fix is a Postgres function (flagged in M-A1).

### [F-004] Founder-update submission — input bounds validation (was part of M-002)
- File: `actions/founder-updates.ts:30-110`
- Problem: The public submit endpoint (token-gated, service-role-bypassed RLS) accepted unbounded text fields and any numeric value. A leaked token allowed inserting megabytes of garbage or NaN/Infinity into the same KPI table the team uses for reporting.
- Fix applied: Added `clampText` (5,000-char cap) on highlights/challenges/next_quarter/ask/notes; `isInvalidMetric` rejecting non-finite, negative, and `> 1e15` values for ARR / revenue / burn / runway / headcount; date-sanity check rejecting unparseable dates and dates more than a year in the future.

### [F-005] `.single()` null-guards in 5 action files (was M-004)
- Files: `actions/option-pools.ts`, `actions/share-series.ts`, `actions/safes.ts` (updateSafe + deleteSafe), `actions/waterfall-scenarios.ts`
- Problem: After `.single()` only `error` was checked. PostgREST returns `data: null` with no error when no row matches, so a deleted-row race or stale client ID would crash the action with `Cannot read properties of null`.
- Fix applied: Replaced `if (fetchErr) …` with `if (fetchErr || !existing) return { error: fetchErr?.message ?? 'X not found' }` at every site.

### [F-006] convertSafe regression tests (was M-005)

- Fix applied: Added 5 tests covering happy path, zero-ownership early-return (no writes), cap-table failure rollback (revert succeeds), cap-table failure rollback (revert also fails — combined error), and missing-SAFE. ### [F-008] deleteStage — track-and-revert compensating move (was M-009)
- File: `actions/pipeline-stages.ts:22-86`
- Problem: When a pipeline stage is deleted, cards are first moved to a fallback stage, then the stage row is deleted. If the delete failed the cards stayed in the fallback column with the source stage still visible (empty). - Fix applied: Capture the affected card IDs first, move only those, and on delete-failure restore only those exact IDs. Updated the surrounding control flow to thread `movedCardIds` cleanly through both branches.

## MANUAL ACTION REQUIRED

### [M-A1] Wrap multi-step server actions in Postgres transactions
- Severity: Critical (financial data integrity — partial mitigation in F-003 and F-008)
- Category: Bug
- Files: `actions/safes.ts` (convertSafe), `actions/pipeline-stages.ts` (deleteStage), and any future multi-write action
- Problem: F-003 and F-008 add best-effort compensating writes, but a true rollback requires a database transaction. The current design is non-atomic — a process crash between the primary write and the compensating revert leaves the database in the same inconsistent state we're trying to avoid.
- Recommended fix: Move the SAFE-conversion and stage-deletion logic into Postgres functions (`SECURITY INVOKER`) wrapped in `BEGIN/COMMIT`. Suggested placement: `supabase/functions/convert_safe.sql` and `supabase/functions/delete_pipeline_stage.sql`. Once in place, simplify the server actions to a single `rpc()` call and remove the compensating-write logic.

### [M-A2] Founder-update token security — schema-dependent hardening
- Severity: High (security)
- Category: Security
- Files: `companies` table, `actions/founder-updates.ts`
- Problem: F-004 added input-bounds validation, but the meaningful security improvements all need DB schema work: (a) `token_expires_at` column so links auto-expire; (b) a `company_update_tokens` side table so rotation has a deprecation window (the current rotation overwrites the column, instantly invalidating any link in flight); (c) a `founder_update_attempts` audit table capturing IP / user-agent / timestamp for every attempt. Without (a) a leaked token works forever; without (b) rotating locks out the founder if their email is delayed; without (c) silent overwrites are invisible.
- Recommended fix: Add the migration with three changes above, then update `submitFounderUpdate` to (1) reject expired tokens, (2) log every attempt to the audit table, (3) consider sending a partner-notification email on every successful submission so silent overwrites are visible.

### [M-A3] RLS uses `authenticated USING (true)` on every table
- Severity: High (security — defence-in-depth)
- Category: Security
- File: `schema.sql:99-105` and any related migrations
- Problem: Every RLS policy is `FOR ALL TO authenticated USING (true) WITH CHECK (true)`. Any logged-in team member can read or write any row, including all financial figures. A session leak gives full write access; there is no audit trail of who edited what.
- Recommended fix: Add a `team_role` enum (`admin`/`partner`/`analyst`/`viewer`); gate destructive operations to `admin`/`partner` via `WITH CHECK (auth.jwt() ->> 'role' IN ('admin','partner'))`. Add `created_by`/`updated_by` columns on financial tables and a write-trigger that populates them. This is the single highest-leverage durability + traceability improvement.

### [M-A4] Token deprecation-window rotation
- Severity: Medium (security UX)
- Category: Security
- File: `actions/founder-updates.ts:90-103` (`generateUpdateToken`)
- Problem: Rotating overwrites `companies.update_token` with a fresh UUID, which instantly stops any link the founder previously received. If the new email is queued or lost, the founder is locked out silently. Rolled into M-A2 above (a side table is the proper fix).

### [M-A5] Detail pages for `/contacts/[id]`, `/pipeline/[id]`, `/tasks/[id]`
- Severity: Low (UX improvement, not a bug — was M-012)
- Category: UI
- Files: `app/(protected)/contacts/`, `app/(protected)/pipeline/`, `app/(protected)/tasks/`; consumers in `app/api/search/route.ts:55,63,71`
- Problem: Companies has a working `[id]` detail page; the other three entities don't. Search results link to the list page, forcing the user to scroll to find the row.
- Recommended fix (option A — full): Add three detail pages mirroring `app/(protected)/companies/[id]/page.tsx` (data fetch + `notFound()` + edit/delete affordances) and update `/api/search/route.ts` to return `/contacts/${c.id}` etc. Substantial new UI surface — needs product input on what each detail page shows.
- Recommended fix (option B — pragmatic): Append `?id=<id>` to search hrefs and add a `useSearchParams` hook to `ContactsClient`, `PipelineBoard`, and the tasks list to scroll-to and highlight the matching row on mount. ~30 lines per file; reuses the existing list UI. Not done in this run because it requires interactive verification on auth-only pages.

## IMPROVEMENT IDEAS

1. **Move `submitFounderUpdate` away from the service-role client, post-migration.** Once M-A2 lands and the token table has its own RLS policies (`token IS NOT NULL AND token_expires_at > now()`), the action can use the anon client and let RLS enforce the capability. That removes one of the two service-role surfaces in the codebase (the other is intentional, in `lib/supabase-server.ts`'s admin path).

2. **Audit log table.** The `authenticated USING (true)` RLS policies will continue to need broad access — the leverage is in *traceability*. A single `audit_log (actor_id, action, table, row_id, before, after, ts)` table with one trigger per financial table costs little and answers "who marked this SAFE converted at 14:03?" forever.

3. **Extend the `inputClasses` pattern beyond `components/forms/`.** The same inline class string (no dark variants) is repeated in `app/login/page.tsx`, `app/mfa/verify/VerifyMFAClient.tsx`, `app/(protected)/settings/security/SecurityClient.tsx`, `app/(protected)/documents/DocumentsVaultClient.tsx`, and `app/updates/[token]/FounderUpdateForm.tsx`. Move `inputStyles.ts` to a shared location (e.g. `lib/form-styles.ts`) and update those five call sites for full dark-mode consistency.

4. **Configure `images.remotePatterns` for Supabase storage in `next.config.ts`.** This unblocks converting the remaining `<img>` tags (company logos in PortfolioTable, GlobalSearch, ReservesClient, CompaniesClient, CompanyForm logo preview) to `next/image`. Needs the production Supabase project ref to construct the host pattern: `'<project-ref>.supabase.co'`.

5. **Smoke-test E2E.** Vitest covers calculations and now `convertSafe` action paths well, but there is no end-to-end check that auth + RLS + middleware compose correctly. One Playwright run nightly hitting `/login`, `/dashboard`, `/companies/[some-id]`, and `/api/cron/lp-report` (with stubbed `CRON_SECRET`) catches drift like a stale `.env` or a redeployed Supabase project.
