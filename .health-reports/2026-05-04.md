## FIXED ISSUES

### [F-001] LP report cron is vulnerable to stored HTML/script injection
- File: app/api/cron/lp-report/route.ts (lines 35–61, new `esc()` helper at 41–46)
- Problem: Company `name` and `sector` fields were interpolated directly into the HTML body of the quarterly LP email without escaping. A team member who creates a company with a name containing `<script>` (or any HTML) would have that markup rendered inside an email sent to *all configured LPs*. The companion route `/api/cron/reminders` already has an `esc()` helper for exactly this reason; the LP report had drifted from that pattern. Severity is high because (a) LPs are the audience and (b) the platform has no input validation that rejects HTML in company names, so this is exploitable today by any authenticated team member with company-create permission.
- Fix applied: Added a local `esc()` helper that HTML-escapes `&`, `<`, `>`, `"`, and applied it to `c.name` and `c.sector` in the table-row template. Sector falls back to `—` when empty (preserves prior visual behavior).

### [F-002] LP report send failures silently reported as success
- File: app/api/cron/lp-report/route.ts (lines 108–123)
- Problem: `await resend.emails.send(...)` discarded its return value. A failed send (bad API key, rate limit, recipient bounce) would still return `{ sent: N, quarter }` to the cron caller, so the cron log would say "delivered" even though no email left the queue. Same class of bug as F-001 in yesterday's report, in a different route.
- Fix applied: Destructured `{ error: sendError }` and return HTTP 500 with the error message if it is truthy.

### [F-003] /api/documents accepted arbitrary file_url, bypassing the storage allowlist
- File: app/api/documents/route.ts (lines 1–35)
- Problem: `/api/extract` and `/api/pipeline/analyze-deck` both call `isAllowedFileUrl()` to ensure that any `file_url` passed in is on the configured Supabase storage host. `/api/documents` (which writes the URL into `documents.file_url`, where it is later fetched and processed by `/api/extract`) had no such check. An authenticated team member could register a document row pointing at any external URL — and that URL would later be fetched server-side by the extraction endpoint, opening the door to SSRF probes against internal/cloud-metadata endpoints from the Vercel function runtime. The allowlist's whole purpose is to be a single boundary; missing the check on the route that *populates the row* defeats the protection.
- Fix applied: Imported `isAllowedFileUrl` and added a 400-with-explanatory-error guard alongside the existing `file_url` validations.

### [F-004] CSV import had no file size limit (DoS / memory exhaustion vector)
- File: app/api/import/csv/route.ts (lines 1–48, new `MAX_UPLOAD_BYTES` constant)
- Problem: `/api/import/csv` accepted arbitrarily large CSV bodies via `formData.get('file')` and called `await file.text()` on whatever was uploaded. Sister route `/api/import` already enforces a 10 MB cap (both via Content-Length pre-check and post-read `file.size` check). The CSV variant did not — a single authenticated user could upload a multi-GB CSV and exhaust the Vercel function's memory.
- Fix applied: Added the same 10 MB ceiling pattern as `/api/import`: Content-Length pre-check returns 413 before reading the body, and a post-read `file.size` check catches clients that omit Content-Length. Documented to match the other route.

### [F-005] Trello import silently dropped pipeline_stages insert errors
- File: app/api/import/trello/route.ts (lines 49–70, errors array hoisted to line 53)
- Problem: Inside the per-list creation loop, `await supabase.from('pipeline_stages').insert({...})` was awaited without destructuring `error`, and the function unconditionally incremented `stagesCreated++`. If the insert failed for any reason (RLS, unique-name collision, schema drift), the response told the caller "we created 5 stages" when the database had received zero. The companion card-import loop in the same function correctly checks `error` and pushes to `errors[]`; the stage loop did not.
- Fix applied: Destructured `{ error: stageErr }`, push to the existing `errors[]` array on failure, only increment `stagesCreated` on success. Hoisted the `errors[]` declaration above its first use (it was previously declared between the stage and card loops).

## MANUAL ACTION REQUIRED

### [M-001] CRON_SECRET appears unset on production — all cron jobs broken
- Severity: Critical
- Category: Bug (Operational)
- File or Source: Vercel project env vars (and `lib/api-auth.ts:39`)
- Problem: A direct GET to `https://menomadin-portfolio.vercel.app/api/cron/reminders` returns HTTP 503 with the body `"CRON_SECRET is not configured on the server"`. This is the explicit fail-closed branch in `requireCronAuth` when `process.env.CRON_SECRET` is undefined. Since Vercel's cron runner reads the same env var to construct the `Authorization: Bearer <secret>` header it sends to the cron route, an unset secret means the runner cannot send a valid header either. All three configured crons in `vercel.json` (`/api/cron/reminders` daily 07:00, `/api/cron/lp-report` quarterly, `/api/cron/scan-emails` daily 08:00) are silently failing — no daily reminder digest, no quarterly LP update, no inbox scanning has run in production since this state began.
- Recommended fix: Generate a new strong secret (e.g. `openssl rand -hex 32`), add it as `CRON_SECRET` in the Vercel project env vars (Production scope), redeploy, and verify by hitting `/api/cron/reminders` with the correct `Authorization: Bearer <secret>` header. Confirm the next scheduled run (07:00 UTC daily) shows `200` and `sent: N` in Vercel cron logs.

### [M-002] /api/send-report uses RESEND_API_KEY as its own bearer token
- Severity: High
- Category: Security
- File: app/api/send-report/route.ts (lines 13–16)
- Problem: The endpoint authenticates incoming requests by comparing the `Authorization` header to `Bearer ${process.env.RESEND_API_KEY}`. The Resend API key is the same secret that the route uses to *send email*; reusing it as the inbound auth token means anyone who legitimately needs the Resend key (a third-party SMTP debugging tool, a logging integration that sees outbound headers) can also call `/api/send-report` and trigger emails to the configured recipients. Two separate secrets — one for outbound to Resend, one for inbound to this endpoint — is the standard pattern. The companion `/api/cron/lp-report` and `/api/cron/reminders` correctly use a dedicated `CRON_SECRET`.
- Recommended fix: Introduce a `REPORT_API_KEY` env var (or reuse `CRON_SECRET` if this endpoint is only invoked by cron). Update the comparison on line 14 and rotate the existing Resend key once the dependency on it for inbound auth is removed.

### [M-003] Cron route (scan-emails) drops insert/update errors on pipeline writes
- Severity: High
- Category: Bug
- File: app/api/cron/scan-emails/route.ts (lines 173–183, 188–191)
- Problem: When the scanner detects a fundraising opportunity, it inserts into `pipeline` and updates `email_integrations.last_scanned_at` without checking the returned `error` on either call. A failed insert (RLS misconfiguration, NOT NULL violation, unique conflict on a duplicate name we didn't filter for) silently increments `totalOpportunities` and is invisible to the caller. A failed `last_scanned_at` update means the *next* cron run reprocesses the same window of inbox messages, potentially producing duplicate pipeline entries (the dedupe is `.ilike(name)`, which doesn't catch slight name-variant duplicates).
- Recommended fix: Destructure `{ error }` on both calls. On insert failure, decrement / don't increment `totalOpportunities` and push to a structured per-integration `errors[]` array returned in the JSON response so cron failures surface in Vercel logs. On `last_scanned_at` update failure, do not advance the `since` cursor for that integration — better to re-scan a window than to permanently skip messages.

### [M-004] Pipeline cron route (analyze-deck) lacks try/catch around Anthropic call

### [M-013] deleteStage moves cards to empty-string status, violating known status constraints
- Severity: Medium
- Category: Bug
- File: actions/pipeline-stages.ts (line 25)
- Problem: When a pipeline stage is deleted, the action runs `supabase.from('pipeline').update({ status: '' }).eq('status', stageName)`. Per the project memory note, `pipeline.status` uses capitalized stage names like "Prospecting" / "Due Diligence" — empty string is not a valid value. Cards moved to `status = ''` will not render under any column on the pipeline board (no column matches an empty string), effectively orphaning them invisibly. The inline comment on line 24 even says "to null / first stage" — the code does neither. Also: the update's `error` is not checked.
- Recommended fix: Either (a) move orphaned cards to the first remaining stage by name (lookup `pipeline_stages` after the delete and pick the lowest `position`), or (b) set status to `null` if the column allows it. In either case, destructure and check `error` on the update before proceeding to delete the stage. If neither stage-move nor null is acceptable, refuse to delete a stage that still contains cards and return a clear error to the UI.

### [M-014] /api/search uses .or() with raw user input — comma in query breaks the filter
- Severity: Medium
- Category: Bug
- File: app/api/search/route.ts (lines 24–26)
- Problem: After escaping `\`, `%`, `_` for ilike, the route passes the pattern into `.or(\`name.ilike.${pattern},email.ilike.${pattern}\`)`. PostgREST's `.or()` syntax separates conditions by commas, so a user query containing a comma (e.g. `acme,inc`) produces a malformed filter string that may match unintended rows or error out. The escape on line 13 protects against ilike wildcards but not against the `.or()` parser's metacharacters.
- Recommended fix: Replace the `.or()` call with two separate queries (one `.ilike('name', pattern)`, one `.ilike('email', pattern)`) and merge results client-side. Alternatively, additionally escape commas, parentheses, and the dot-separator in the pattern, but two queries is simpler and harder to get wrong.

### [M-015] usePermissions is a stub that grants all permissions
- Severity: Low
- Category: Code Quality
- File: hooks/usePermissions.ts (lines 1–15)
- Problem: The hook returns `{ allowed: true }` for every action and uses `any` types on its parameters. The file comment explains "permissions system removed" — which is fine for an internal tool that relies on Supabase RLS for authorization. But components throughout the UI (`CompaniesClient.tsx`, `companies/[id]/CompanyDetailClient.tsx`, etc.) gate destructive UI affordances behind these checks under the assumption that they meaningfully discriminate. With the stub in place the gating is theatrical, and the `any` types disable any future help from the type checker if/when permissions return.
- Recommended fix: Either (a) remove the gating from consumers and delete the hook (RLS will deny destructive actions server-side anyway), or (b) replace the `any` parameters with the real action/table types so the call sites stay honest about what they're checking, even while the implementation is permissive. Option (a) reduces dead code; option (b) preserves the migration path back to a real permissions model.

### [M-016] CSV parser does not handle escaped double-quotes inside fields
- Severity: Low
- Category: Bug
- File: app/api/import/csv/route.ts (lines 9–20)
- Problem: The hand-rolled `splitLine` toggles `inQ` on every `"` it encounters, so the standard CSV escape `""` (a literal double-quote inside a quoted field, per RFC 4180) is mis-parsed: the first `"` exits the quoted state, the second `"` re-enters it, and any commas between are not treated as separators. A field like `"Smith, ""Bob"" Co."` will produce wrong column splits.
- Recommended fix: Replace with `papaparse` (already a transitive dependency in the React ecosystem) or write the proper RFC 4180 escape: when inside a quoted field and the current char is `"`, peek ahead — if the next char is also `"`, append a single `"` and skip both.

## IMPROVEMENT IDEAS

1. **Set up an alert for cron-route 503s.** The fact that all three cron jobs have been failing closed in production with no one noticing (M-001) is the kind of operational gap that justifies a Vercel monitoring rule on `/api/cron/*` non-200 responses. Even a simple `vercel logs --filter status=503` cron-of-the-cron would catch this class of issue.

2. **Centralize HTML escaping in a shared helper.** Both `lib/api-auth.ts`-adjacent cron routes (reminders, lp-report) had to define their own `esc()` function — and the LP report drifted (F-001). A `lib/html-utils.ts` exporting a single `escapeHtml()` (and ideally enforced via lint rule on any string interpolated into an HTML template literal) eliminates that drift class.

3. **Wrap mutating Supabase calls in a `mutate()` helper that throws.** Issues M-007, M-008, M-011, M-012, M-013, F-002, F-005 are the same pattern: someone forgot to destructure `error` from a mutation and silently dropped it. A `mutate(qb)` wrapper that throws on `.error` (and that the linter requires for `.insert/.update/.delete/.upsert` calls) would systematically eliminate this class of bug.

4. **Add a "production health" admin page that runs pre-flight checks.** A `/admin/health` route that exercises `CRON_SECRET` presence, validates that all three cron schedules in `vercel.json` map to existing routes, that `RESEND_API_KEY` and `LP_EMAILS` are non-empty, and that the Anthropic API key works — surfaced behind an admin-only check — would convert silent operational failures (M-001) into something a fund manager glances at once a week.

5. **Stop reusing the Resend API key as an auth token (M-002), then audit all `process.env.*` lookups for similar dual-purpose patterns.** Once the immediate fix for M-002 lands, a quick repo-wide grep for env vars used in `Authorization` comparisons would catch any other accidental "secret = service-key" reuse before it becomes an incident.
